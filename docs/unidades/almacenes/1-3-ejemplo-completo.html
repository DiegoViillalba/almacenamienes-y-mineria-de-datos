Ejemplo Completo: Café Norte — Análisis y Construcción de Data Marts (Kimball)

3.1 Propósito y supuestos

Este ejemplo documenta, con un enfoque Kimball, la construcción de dimensiones conformadas y una tabla de hechos a granularidad línea de pedido para evaluar el impacto del lanzamiento de la app (enero de 2023) y del programa de lealtad.

Hito de intervención (cutoff): 2023-01-01.
Se define:

  • Periodo pre: fechas menores a 2023-01-01.
  • Periodo post: fechas mayores o iguales a 2023-01-01.

Se asume que el lector ya cuenta con CSVs provenientes de sistemas operacionales (POS y App) y catálogos maestros.

3.2 1. Estructura esperada de archivos CSV

Coloque los archivos en un directorio data/ al mismo nivel que este .qmd.

3.2.1 1.1 Catálogos maestros (dimensiones)

  • data/clientes.csv
    Campos mínimos: cliente_id, fecha_alta, tipo_cliente, nivel_lealtad (u otros atributos).

  • data/productos.csv
    Campos mínimos: sku, nombre, categoria, subcategoria, tamano, activo.

  • data/sucursales.csv
    Campos mínimos: sucursal_id, nombre, zona, formato.

  • (Opcional) data/promociones.csv
    Campos mínimos: promo_codigo, campania, descripcion.

3.2.2 1.2 Transacciones (hechos)

Se recomienda conservar el encabezado del pedido y las líneas separadas.

  • data/pedidos_pos.csv y data/pedidos_app.csv
    Campos mínimos: order_id, fecha, cliente_id, sucursal_id, canal (POS o APP).

  • data/lineas_pos.csv y data/lineas_app.csv
    Campos mínimos: order_id, line_num, sku, cantidad, precio_unitario, promo_codigo, metodo_pago, cancelado_flag.

Nota: si en origen el descuento ya viene desglosado, incluya también descuento_importe.
Si no, puede derivarse en el proceso (ver más adelante).

3.3 2. Carga de datos y validación de esquema

En un pipeline reproducible, la primera etapa consiste en cargar datos y aplicar validaciones: tipos, llaves no nulas, dominios controlados y rangos.

3.3.1 2.1 Chequeos mínimos de calidad

A continuación se implementan reglas mínimas. En un DW real, estas reglas alimentan un sistema de monitoreo (por ejemplo, Great Expectations) y generan reportes de calidad.

Ver código 
def assert_columns(df: pd.DataFrame, required: list[str], name: str):
    missing = [c for c in required if c not in df.columns]
    if missing:
        raise ValueError(f"[{name}] Faltan columnas requeridas: {missing}")

# Esquema mínimo
assert_columns(clientes,  ["cliente_id", "fecha_alta"], "clientes")
assert_columns(productos, ["sku", "nombre", "categoria"], "productos")
assert_columns(sucursales,["sucursal_id", "nombre", "zona"], "sucursales")

assert_columns(ped_pos, ["order_id","fecha","cliente_id","sucursal_id","canal"], "pedidos_pos")
assert_columns(ped_app, ["order_id","fecha","cliente_id","sucursal_id","canal"], "pedidos_app")

assert_columns(lin_pos, ["order_id","line_num","sku","cantidad","precio_unitario"], "lineas_pos")
assert_columns(lin_app, ["order_id","line_num","sku","cantidad","precio_unitario"], "lineas_app")

# Tipos: fechas
for df, col, name in [
    (clientes, "fecha_alta", "clientes"),
    (ped_pos, "fecha", "pedidos_pos"),
    (ped_app, "fecha", "pedidos_app"),
]:
    df[col] = pd.to_datetime(df[col], errors="coerce")
    if df[col].isna().any():
        bad = df[df[col].isna()].head(5)
        raise ValueError(f"[{name}] Fechas inválidas en '{col}'. Ejemplos:\n{bad}")

# No nulos en llaves operacionales
def assert_not_null(df: pd.DataFrame, cols: list[str], name: str):
    for c in cols:
        if df[c].isna().any():
            raise ValueError(f"[{name}] Hay nulos en la columna clave: {c}")

assert_not_null(ped_pos, ["order_id","cliente_id","sucursal_id"], "pedidos_pos")
assert_not_null(ped_app, ["order_id","cliente_id","sucursal_id"], "pedidos_app")
assert_not_null(lin_pos, ["order_id","sku"], "lineas_pos")
assert_not_null(lin_app, ["order_id","sku"], "lineas_app")

# Dominios controlados
canales_validos = {"POS","APP"}
if not set(ped_pos["canal"].unique()).issubset(canales_validos):
    raise ValueError(f"[pedidos_pos] Canal fuera de dominio: {set(ped_pos['canal'].unique()) - canales_validos}")
if not set(ped_app["canal"].unique()).issubset(canales_validos):
    raise ValueError(f"[pedidos_app] Canal fuera de dominio: {set(ped_app['canal'].unique()) - canales_validos}")

# Rangos numéricos
for df, name in [(lin_pos,"lineas_pos"), (lin_app,"lineas_app")]:
    if (df["cantidad"] <= 0).any():
        raise ValueError(f"[{name}] Se encontraron cantidades <= 0")
    if (df["precio_unitario"] < 0).any():
        raise ValueError(f"[{name}] Se encontraron precios negativos")

print("Validaciones mínimas: OK")

3.9 8. Discusión técnica: decisiones de diseño y extensiones

  1. Dimensiones conformadas: dim_tiempo, dim_cliente, dim_producto, dim_sucursal, dim_canal permiten integrar Ventas POS y Pedidos App en una misma semántica de consulta (bus architecture).

  2. Granularidad: al modelar a nivel de línea, se preserva el análisis de mezcla de productos y el efecto de promociones en el importe.

  3. SCD Tipo 2 (extensión recomendada): si se desea preservar historia de cambios en categoria o en nivel_lealtad, se debe:

    • añadir vigencia_inicio, vigencia_fin, vigente_flag,
    • resolver la clave dimensional “as-of” en la fecha del hecho.

  4. Late Arriving Dimensions: en escenarios reales, pueden llegar transacciones cuyo cliente aún no existe en el catálogo. La práctica estándar es asignar Unknown Member (*_key = 0) y corregir al cierre del ciclo de carga.

  5. Prueba estadística pre/post (opcional): para un tratamiento académico, se sugiere formalizar el cambio con pruebas no paramétricas o bootstrap sobre series mensuales (p. ej., pedidos o tasa de cancelación).

3.10 9. Reproducibilidad y control de versiones

Para un libro académico, se recomienda:

  • fijar versiones de dependencias (p. ej., pandas==2.x, numpy==2.x),
  • mantener data/ como directorio externo (no versionar datos sensibles),
  • versionar el pipeline y el .qmd (Git),
  • documentar el diccionario de datos final (metadatos de cada dimensión y hecho).
